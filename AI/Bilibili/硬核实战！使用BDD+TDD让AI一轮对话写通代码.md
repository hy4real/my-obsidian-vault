# 硬核实战！使用BDD+TDD让AI一轮对话写通代码

> [!info] 视频信息
> 硅基包工头 / 04:50 / 2026-05-08 / [链接](https://www.bilibili.com/video/BV1NpRmBLEbp/)

## 核心观点

在 AI 编程时代，BDD+TDD 的组合能让模型通过测试用例不断自我验证，实现「需求明确后一轮对话写通代码」。关键不在于照搬传统流程，而在于理解方法论的本质——BDD 用于标准化需求描述（模型天然理解），TDD 用于给模型一个可执行的验收标准。

## 前置准备

**TDD Skill 安装**：从 Superpowers 的 GitHub 源码中拷贝 TDD Skill 到项目 `.claude/skills/` 目录下。也可直接安装 Superpowers 插件（`/plugin` 搜索安装）。

**BDD 配置**：反直觉的做法——不用任何 Skill，只在 CLAUDE.md 加两行规则，让模型将 BDD 用例描述作为注释写到代码文件里。传统 BDD 流程（feature 文件 + 框架）是为产品经理和程序员协作设计的，单人开发时不需要那套流程，只需要利用 BDD 的标准化描述方式（模型看到 BDD 缩写就知道怎么做）。

## 实战流程

1. **需求**：为 Godot 美少女战棋游戏增加「毒蛇随从」+「中毒机制」（每回合结束掉 1 点血）
2. **BDD 用例**：模型先写文字用例 → 发现「中毒状态可叠加」描述不准确 → 修正为不可叠加
3. **TDD 测试**：引用 TDD Skill，先写测试用例（RED）
4. **实现代码**：模型自动实现 → 多次迭代修复错误 → 最终全部通过（GREEN）
5. **验证**：实际运行游戏，功能正确

演示使用 MiniMax M2.7（白菜价模型），刻意展示容错性——模型需要多次修改才通过测试，但最终自动完成。

## 适用边界

| 场景 | 适合度 | 原因 |
|------|--------|------|
| 可定义接口的业务逻辑 | 高 | 测试用例清晰，模型可自我验证 |
| 有独立状态层的代码 | 高 | 状态可脱离 UI 单独测试 |
| 图形界面开发 | 低 | UI 测试痛苦且不稳定 |
| 代码结构不可测 | 低 | 无法写有效测试用例 |

**核心洞察**：TDD 的一个优势是「能测试的代码结构天然是好结构」。为了让代码可测，需要将业务逻辑从 UI 中剥离出独立的状态层。

## 个人思考

- BDD 在单人 AI 编程中的价值是标准化需求描述，而非流程本身——这是一个「知其所以然」的好例子
- 测试用例作为模型的「验收标准」，比自然语言需求更精确，减少反复沟通

- [ ] 在自己的项目中实践：先 BDD 描述用例注释 → 再 TDD 测试 → 再实现
