# 十年老技术开发的 AI Agent 探索之路

> 一个在"AI 娱乐圈"摸爬滚打的老开发，试图从月抛式的焦虑中找到不会过期的东西。

---

## 一句话总结

Agent 的价值不是替人做事，而是把依赖人的高频工作改造成可执行、可观测、可复盘、可优化的系统——**让人退出微观调度**。

---

## 第一章：人是瓶颈，但不是替代对象

同时开 4-6 个 AI 终端是人脑并发的硬上限。人工并发有三个硬伤：
- **吞吐有限**：一天能管的 Agent 窗口有硬上限
- **状态不稳定**：上下文丢失、判断漂移、质量波动
- **难以规模化**：做成一次不难，稳定重复难

**核心洞察**：解决瓶颈的方式不是让 AI 替代人，而是让系统不再依赖人的实时在场。

---

## 第二章：80% 的 AI 需求不需要 AI

**原则：代码优先于 Prompt，能用 10 行 Bash 解决的，别折腾 AI。**

决策层级（不确定性递增）：

| 层级 | 适用场景 | 不确定性 |
|------|---------|---------|
| 目标层 | 想清楚到底要解决什么 | 最低 |
| 代码层 | if/else、正则、模板引擎 | 低 |
| CLI 层 | grep + jq + curl 串流程 | 中低 |
| Prompt 层 | 需求翻译、文案生成 | 中高 |
| Agent 层 | 多步推理、动态决策 | 最高 |

> 每往上一层，不确定性和成本都增加一个量级。**能在下层解决的，绝不上推。**

---

## 第三章：Vibe Coding 翻车记

| 阶段 | 状态 |
|------|------|
| Day 1-3 | ✨ 成就感爆棚 |
| Day 7 | ⚠️ 打地鼠式 bug |
| Day 14 | 🔥 过度设计、冗余逻辑泛滥 |
| Day 15 | 🔧 花一整天做"设计与实现对齐" |

**本质：Vibe Coding 是先易后难**。前期省掉的设计时间，后期以 10 倍 debug 时间还回来。

**SDD（Spec-Driven Development）恰好相反——先难后易。** Day 15 的阵痛建立了让系统后续能自动运转的全部基础。

---

## 第四章：24h 打工人——第一个真正的系统

### 核心架构：文件 + 轮询

```python
class AgentScheduler:
    def run(self):
        while True:
            task = self.queue.poll()
            tool = self.tools.get_available()  # 自动选可用工具
            result = tool.execute(task.prompt, task.workspace)
            # 失败时自动冷却并换工具重试
```

调度层四件事：接收任务、分发执行、状态管理、失败切换。

### SDD：让每一步都有据可查

```
feedbacks/
├── sdd/
│   ├── spec.md      # 规格说明：目标、验收标准
│   ├── plan.md      # 技术方案：涉及文件、实现步骤
│   └── tasks.md     # 任务清单
├── tasks.json       # 任务执行状态
└── debug/           # prompts + agent.log
```

> **留痕不是为了 debug，而是为了进化。**

### 智能并发策略

| 策略 | 理由 |
|------|------|
| 组间并发（前端/后端并行） | 代码在不同目录，不冲突 |
| 组内串行 | 避免同文件冲突 |
| 失败隔离 | 故障不扩散 |

### 工具失败自动切换

codex 配额耗尽 → gemini-cli → claude → 等待 5 分钟后自动探活。

## 第五章：Agent 自己修了自己的 bug

系统通过自己的反馈系统提交了一个 bug，自动走完 SDD 流程（spec → plan → task → 修复），全程只人工确认了澄清结果。

**自举的三个前提：**
1. 清晰的设计文档
2. SDD 标准流程
3. **constitution.md**——架构约束文件，定义代码组织规范、命名规则、模块边界

## 第六章：从 demo 到系统——门槛是治理

### Observability：6 个必看维度

| 维度 | 看什么 |
|------|--------|
| 1_goal | 当前目标是什么 |
| 2_step | 正在执行哪一步 |
| 3_tool | 用了什么工具 |
| 4_failure | 为什么失败 |
| 5_recovery | 是否触发重试/回退/切换 |
| 6_cost | 本轮 token/时间/成本 |

### Eval：4 个持续校准问题

- 需求澄清是否稳定？
- 任务拆解是否越来越合理？
- 某个 Skill 是否真的提高了成功率？
- 失败切换有没有制造新错误？

### 核心判断：脚手架 > 模型

| 投入 | 成本 | 效果 |
|------|------|------|
| 模型升级 | +300% | +20% |
| 脚手架升级 | +50% | +200% |

> 垃圾的思考乘以强大的模型，等于精美的垃圾。

## 第七章：协议层正在成形

Agent 开发正在从"框架之争"转向"协议 + runtime + control plane 之争"。

| 时间 | 事件 | 意义 |
|------|------|------|
| 2025-03-11 | OpenAI 发布 Agents 新基座 | Responses API runtime 收敛 |
| 2025-04-09 | Google 发布 A2A | Agent 间协作协议化 |
| 2025-06-23 | A2A 捐给 Linux Foundation | 变行业标准 |
| 2026-02-13 | GitHub Agentic Workflows | Agent 进入 CI/PR/Issue 主流程 |
| 2026-03-16 | Microsoft Foundry Agent Service GA | 全链路 tracing + eval |
| 2026-04-07 | GitHub Dependabot → AI agent | 安全告警自动修复 |

**三大协议层：**
- **Responses API**：runtime 收敛
- **MCP**：工具接入标准化
- **A2A**：多 Agent 协作协议

> 未来做 Agent，越来越像搭操作系统，不只是写 prompt。

## 第八章：从 Task-Driven 到 Goal-Driven

| 维度 | Task-Driven | Goal-Driven |
|------|-------------|-------------|
| 人的角色 | 项目经理 + 执行监督 | 目标设定者 / 审核者 |
| Agent 的角色 | 执行器 | 自主推进者 |
| 决策中心 | 在人脑子里 | 在目标 + 边界 + 系统状态里 |
| 适用场景 | 简单、一次性任务 | 长期、复杂、持续推进任务 |

> Task-Driven 解决执行问题，Goal-Driven 解决迭代问题。

### Goal-Driven 的 5 个前提

1. 目标必须清晰（可推进、可判断）
2. 边界必须清晰（能做什么、资源上限）
3. 状态必须可见
4. 过程必须留痕
5. 权限必须可控

> **Goal-Driven 不是更放权，而是更强约束下的有限自治。**

### 共享状态：STATE.yaml

每个 Agent 自己读取状态、写回进度，主会话只负责高层目标和验收。

### 6 步落地路径

1. 写清楚 spec（要做什么、不做什么、怎么算完成）
2. 执行过程留痕（Prompt/状态/输出/错误全记录）
3. 补 observability 和 eval（知道为什么成功/失败）
4. 高频动作沉淀为 Skill
5. 引入调度和并发
6. 最后才尝试 Goal-Driven

> **先让一次执行可复盘，再让它可重复，再让它可规模化，最后让它可有限自主。**
