### 第二周（3月23-29日）

**模型微调基础工作——环境、数据、脚本**

#### 阶段1：技术栈确定
- **初期尝试:** Qwen3.5-35B-A3B（4bit 量化）
  - 问题：4x RTX4090 上仍无法满足 Unsloth 推荐配置
  - 公司 A100 被占用，替代方案受限
  - **决策:** 降配到 Qwen3.5-9B

- **最终选择:** Qwen3.5-9B + Unsloth + bf16 LoRA
  - 理由：显存可控、效果有保证、LlamaFactory 被否

#### 阶段2：冒烟测试
- **规模:** 50 条样本，2 张 4090 卡
- **结果:** ✅ 环境验证通过，无依赖冲突

#### 阶段3：数据工程（核心工作）
**输入:** 49 篇聚合物论文 + 人工审核的 JSON 配对数据

**问题发现与修复:**
1. **0数据切片:** 23 篇论文初期无有效切片 → 排查后全部修复
2. **JSON 格式混乱:** 发现 3 种不同的数据封装格式
   - 标准格式
   - list 包裹格式
   - output 包裹格式
3. **字段类型不一致:** 位置索引既有 dict、list，也有 str
4. **语言混杂:** 性质类别中英文混用

**最终产出:**
- 🎯 **3169 条高质量训练切片**
- 📊 按文献级别 8:1:1 比例切分（训练/验证/测试）

#### 阶段4：训练脚本开发
```
特性：
- train_on_responses_only：仅在 JSON 输出部分计算 loss
- eval_loss 监控：实时追踪验证集表现
- 完整的 logging 和 checkpoint 保存机制
```

**核心发现:** 这周全流程都由 Claude 辅助完成，积累了完整的 Claude Code project

### 第三周（3月30-4月5日）

**数据质量危机 → 标准化优先策略转变**

#### 问题揭示
**发现:** Agent（包括 LLM）为追求输出效果，会自行"作弊"
- 表现形式：生成虚假数据、篡改标签、隐瞒问题
- 影响：模型在此数据上微调后，泛化性破裂

#### 战略调整
**关键认识:**
> 数据质量 > 模型复杂度  
> 前期应该定好**严谨的标准和基线**，而非盲目追求指标

**方向确认:**
1. ✅ 制定数据标准（而非仅有定量指标）
2. ✅ 基线模型在高质量数据上跑通
3. ⏳ 后续再用 autoresearch 优化超参、探索方案空间

#### 下周规划（多Agent对抗）
提出假说：**能否通过两个独立Agent的对抗机制，互相审核和改进？**

### 第四周（4月6-13日）

**多Agent对抗机制实践 + 工程基础补强**

#### 核心实验：两个Agent的对抗评审系统

**架构设计:**
```
Agent A (执行者)：GLM-5-Turbo
  ├─ 职责：修改代码、运行实验、分析结果
  ├─ Session：独立，不共享上下文
  └─ 输出：experiment report + code changes

Agent B (审阅者)：GLM-5.1  
  ├─ 职责：审核 A 的结果、提出建议、规划下步
  ├─ 上下文：仅通过读取仓库文件（A 的 report）
  └─ 输出：review + improvement plan
```

**执行方式:**
- 两个 Claude CLI 会话，完全隔离
- 通过**共享代码仓库**作为唯一通信媒介
- 避免上下文污染，逼迫 Agent 基于事实判断而非记忆

**初步成果:** 
- 🎯 有效提升了问题发现和修复质量
- 📈 A 的执行更谨慎，B 的建议更中肯

#### 遇到的问题

| 问题 | 现象 | 根因分析 |
|-----|------|--------|
| 频繁权限确认 | "Do you want to proceed?" 弹窗 | 智谱模型集成策略（非原生 Opus/Sonnet）|
| 工具适配 | 部分工具使用体验不一致 | 多模型后端差异 |

#### Agent 对抗机制反思

本周的两 Agent 对抗机制验证了一个关键假设：**通过完全隔离的上下文，可以让 Agent 基于事实而非记忆做决策**。这套机制的可复用性高，适合在团队内推广用于 code review 和决策评审。
