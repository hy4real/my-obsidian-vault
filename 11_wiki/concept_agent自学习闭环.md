# Agent 自学习闭环

> **核心观点**：Agent 不是一次性回答机器，而是一个能把有效经验提炼、写回、再利用的系统。

## 核心定义

Agent 自学习闭环，是指 Agent 在执行任务过程中，通过明确的触发器进入复盘，把经验拆成可持久化资产，再在下一轮执行中重新注入，从而形成持续提升的学习回路。

```
触发 → 复盘 → 写回 → 再注入 → 再执行
```

这个闭环的关键，不是“记录过”，而是“下一轮真的还能用上”。

## 四个环节

### 1. 触发

- 按对话轮次触发 `memory review`
- 按工具调用次数触发 `skill review`
- 关键不是频率本身，而是**明确何时进入学习阶段**

### 2. 复盘

- `memory review` 看偏好、事实、长期上下文
- `skill review` 看非平凡方法、试错过程、可复用流程
- 目标不是泛泛总结，而是结构化提炼

### 3. 写回

- `memory` 写入用户偏好、环境事实等声明性知识
- `skill` 写入可复用方法、策略和步骤
- `session` 保留过程痕迹，必要时可回溯

### 4. 再注入

- 下一轮 prompt 读取已沉淀内容
- 让历史经验真正影响后续决策
- 没有再注入，就只是做了记录，不是闭环

## 设计原则

- 复盘必须后置，不能阻塞主任务
- 长期资产要分层存放，避免知识库变脏
- 写回结果必须可检索、可引用、可再利用
- 闭环的价值来自复利，而不是单次总结

## Hermes 的实现提示

这篇文章里，Hermes 给出了一个比较完整的工程样板：

- `Memory Nudge`：默认每 10 轮对话触发一次
- `Skill Nudge`：默认每 15 次工具调用触发一次
- `spawn_background_review`：在后台异步复盘，不阻塞主对话
- `memory` / `skill` 分层写回：把事实和方法分开存
- `reinjection`：让下一轮 prompt 重新读到这些结果

这说明自学习闭环不是抽象口号，而是可以落成一条稳定的工程链路。

## 与知识工程的对应

- `memory` 对应知识库里的稳定事实
- `skill` 对应可复用的操作方法
- `background review` 对应异步 ingest / lint 风格的维护流程
- `reinjection` 对应把整理后的知识重新写入工作流

这也是为什么它和 [[concept_wiki系统]] 天然相关——Wiki 的 Ingest/Query/Lint 就是自学习闭环的知识管理版本。
