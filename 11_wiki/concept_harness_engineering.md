# Harness Engineering — 引导 AI 而非增强 AI

> 工作流只是管道，知识才是流过管道的活水。  
> Skill、Agent、工具链会随模型迭代更新，但领域知识是永恒的。  
> — stevenpxiao，腾讯技术工程，2026-04-27

---

## 核心定义

**Harness Engineering**（马具工程）：像骑师通过缰绳引导马的力量，而非增强马本身体能——强调**引导和约束 AI 模型**，而非提升模型本身。

业界三大标志性实践：

| 实践方 | 核心关注 | 关键动作 |
|--------|---------|---------|
| OpenAI — Codex | 人机交互协议 | 零手写代码，通过精确指令驾驭 Agent |
| Cursor — Self-Driving | 多 Agent 协同 | 背景 Agent 自动检测冲突并运行测试 |
| Anthropic — Claude Code | 长时运行稳定性 | 多层记忆 + CLAUDE.md 约束，复杂任务保持一致性 |

```
┌─────────────────────────────────────────────────────┐
│              Harness Engineering 三支柱              │
├─────────────────┬─────────────────┬─────────────────┤
│  上下文工程      │  架构约束        │  持续治理        │
│  Context Eng.   │  Architecture    │  Governance      │
├─────────────────┼─────────────────┼─────────────────┤
│ · 长/短期记忆   │ · Agent 编排模式 │ · 质量门禁       │
│ · 知识检索注入  │ · 状态机设计     │ · 知识生命周期   │
│ · 渐进式披露    │ · 降级策略       │ · 自动衰减       │
│ · 上下文防火墙  │ · 安全边界       │ · 持续进化       │
└─────────────────┴─────────────────┴─────────────────┘
```

**关键洞察**：知识管理（检索注入、知识生命周期）是 Harness 的核心能力，不是附属品。业界讨论偏重"工作流编排""Agent 协同"，忽略了底层知识基础设施。

## 核心论点：为什么知识比工作流更重要

### 1. 工作流"可替换"，知识"可累积"

工作流会演化（状态机→DAG→…），Agent 调度从串行到并行到分层级联。模型厂商会内化工作流规划能力。

但领域知识不会过时："高并发预算扣减需要 Redis+Lua 保证原子性"——这条知识不管工作流怎么变，都有价值。

### 2. 没有知识沉淀的工作流是"一次性"的

反模式：复杂 Agent 工作流每次从零开始——上次踩的坑下次照踩，上次的架构决策下次重新推导。投入工程成本搭工具链，却没让工具链变得越来越聪明。

### 3. 知识是团队的"复利资产"

| 知识类别 | 特点 |
|---------|------|
| 散点型 | 孤立的事实，易获取 |
| 因果型 | A 导致 B 的推理链，模型难以提供 |
| 时空型 | 特定场景/时间窗口才成立，最依赖实践积累 |

越高阶的知识，越难从模型获得，越依赖团队实践积累。知识库有成百上千条 proven 条目时，新成员、新项目都能"站在前人肩上"——这是复利效应。

## 知识流动的工作流设计

工作流三阶段服务于知识：

```
INIT（知识注入）
→ 工作流启动时 git pull 团队知识仓库，注入全景目录
→ 新工作流自动站在前人肩上

各阶段执行中（知识消费）
→ Agent 在决策点按需查询知识库（三级渐进式索引）
→ 每个阶段有独立查询预算，聚焦特定类型

ARCHIVE（知识提取）
→ 工作流完成后自动提取：决策→decision，踩坑→pitfall，经验→guideline
→ 符合条件的自动提升到团队知识库
```

各阶段查询焦点：

| 阶段 | 重点知识类型 |
|------|------------|
| 产品分析 | model, process, pitfall |
| 技术分析 | decision, guideline(avoid), pitfall |
| 架构设计 | decision, model |
| 编码实现 | guideline, pitfall |
| 验证构建 | pitfall, guideline(avoid) |

## Big Model vs Big Harness — 务实立场

这不是非此即彼的选择：

- **模型能力提升**是大势所趋，知识架构应对模型升级保持开放（更强的模型能更好利用同样的知识）
- **模型不能替代领域知识**：再强的模型也不知道你的业务系统里有哪些隐藏坑
- **知识工程投入是确定性回报**：每沉淀一条 proven 知识，所有后续工作流受益；模型提升的回报是概率性的

---

## 人机交互瓶颈

传统工作流隐含假设"人随时在工位"，但一天 8 小时工作真正能操控 Agent 的不到 4 小时。

解法：**异步审批架构**
- 状态持久化（文件系统即状态机，不依赖内存）
- 断点恢复（每阶段有持久化产物，任意断点恢复）
- 异步审批（Agent 提交产物等待，人可在任意时间任意设备审批）
- 通知触达（关键节点主动推送，而非被动等人来检查）

目标：工作流 7×24 小时顺畅流转，碎片时间（会议间隙、通勤）都能推进工作流，加速知识沉淀闭环。
