# LLM-Wiki 知识管理系统

> 不要把 LLM 当搜索引擎，让它像程序员一样**持续维护一个 Markdown 知识库**。 — Karpathy

## 核心理念

用户负责找资料和提问，LLM 负责总结、交叉引用、分类整理、维护一致性。知识库随时间持续增长，形成可复利增长的资产。

传统 RAG 每次提问从零开始，答案用完即弃。LLM-Wiki 让知识**编译一次、持续复用**。

| 维度 | 传统 RAG | LLM-Wiki |
|------|---------|----------|
| 知识处理 | 每次临时检索 | 录入时预先编译 |
| 积累效应 | 无，每次从零 | 持续复利增长 |
| 交叉引用 | 无 | 丰富，自动维护 |
| 矛盾检测 | 无 | Lint 自动发现 |
| 维护负担 | 低（被动） | 低（LLM 承担） |

## 三层架构

```
┌─────────────────────────────────────┐
│ 第三层：Schema（CLAUDE.md）         │
│ → 告诉 LLM 如何组织、约定、流程    │
├─────────────────────────────────────┤
│ 第二层：Wiki（11_wiki/）            │
│ → LLM 写，用户读；结构化链接网络   │
├─────────────────────────────────────┤
│ 第一层：Raw Sources（00_raw/）      │
│ → 只读，LLM 不可修改；原始数据来源 │
└─────────────────────────────────────┘
```

- **第一层**不可变，保证可追溯
- **第二层**由 LLM 主导维护，用户通过 Obsidian 浏览
- **第三层**由用户定义，约束 LLM 行为

## 三大操作

| 操作 | 触发 | LLM 做什么 |
|------|------|-----------|
| [[guide_ingest操作|Ingest]] | 丢入新资料 | 提取概念 → 创建页面 → 更新链接 |
| [[guide_query操作|Query]] | 对 Wiki 提问 | 搜索 → 综合 → 问要不要存 |
| [[guide_lint操作|Lint]] | 定期检查 | 找死链、孤儿、矛盾 → 修复 |

一个来源的 Ingest 可能牵动 10-15 个页面的更新。好的 Query 答案可以回存 Wiki，成为新页面。Lint 保证网络健康。

三者形成飞轮：Ingest 增加节点 → Query 发现关联 → Lint 强化网络 → 知识复利增长。

---

## 知识复利

传统学习线性积累（+1, +2, +3），Wiki 模式指数积累——每新增一个节点都与现有节点产生连接，价值呈指数增长。

人类维护 Wiki 最终会因为琐碎簿记工作（更新交叉引用、保持摘要最新、标注矛盾）而放弃。LLM 不会厌倦，维护成本接近零，让复利曲线可持续。

历史渊源：1945 年 Vannevar Bush 在《As We May Think》中设想了 Memex——个人化知识存储，文档关联和文档本身同样有价值。LLM 解决了"谁来维护"的问题。

## LLM 作为知识工程师

| 任务 | 人类 | LLM |
|------|------|-----|
| 阅读和思考 | 擅长 | 擅长 |
| 更新交叉引用 | 费时费力 | 秒级完成 |
| 检测矛盾 | 容易遗漏 | 全量扫描 |
| 批量修改 15 个文件 | 很难 | 一次完成 |
| 长期坚持 | 最终放弃 | 永不疲倦 |

角色隐喻：Obsidian 是 IDE，LLM 是程序员，Wiki 是代码库，用户是产品经理，Schema 是架构设计文档。

## 工具生态

- **sage-wiki**（Go）：增量编译、搜索、问答，可作 MCP Server
- **Claude Code Skill**：一行命令装上
- **Obsidian Web Clipper**：网页转 Markdown
- **Dataview 插件**：基于元数据动态查询
- **图谱视图**：看 Wiki 整体形态

在中等规模（~100 来源、几百页面）下，直接用索引文件就够，无需向量数据库。
