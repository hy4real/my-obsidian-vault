# 字节Data Agent岗面试常见题

基于四面字节Data Agent岗的真实面试经历整理，覆盖Agent、RAG、评测、微调四大核心模块。

---

## 一、Agent架构（高频）

### 1. 为什么自己搭Agent不用现成框架？

**答案**：

现成框架（LangChain、LlamaIndex）封装太深，调试困难。业务场景有定制需求，比如工具调用逻辑、上下文管理策略，自己搭更灵活可控。

关键点是理解Agent的本质：**感知-推理-行动循环**。不管用什么框架，核心都是：
1. 接收用户输入
2. 模型推理（决定调用什么工具）
3. 执行工具，获取结果
4. 把结果塞回上下文，继续推理

自己搭的好处是每一步都能控制，出问题能快速定位。

### 2. ReAct是什么？和Function Call有什么区别？

**答案**：

**ReAct（Reasoning + Acting）**：
- 提示词层面的推理模式
- 模型先输出"思考"（Thought），再输出"行动"（Action），然后观察结果（Observation）
- 循环执行直到得到最终答案

**Function Call**：
- 模型层面的工具调用能力
- 模型直接输出结构化的JSON，指定要调用的函数和参数
- 由外部系统执行，结果返回给模型

**核心区别**：
- ReAct是"提示词工程"，靠提示词引导模型按格式输出
- Function Call是"模型能力"，模型原生支持结构化输出
- Function Call更稳定（不容易格式错误），ReAct更通用（不依赖模型支持）

### 3. 模型不支持Function Call怎么办？

**答案**：

手搓一个。核心思路是用提示词模拟：

1. 在System Prompt里描述可用工具（名称、参数、功能）
2. 要求模型输出JSON格式的工具调用请求
3. 自己解析模型输出，提取工具名和参数
4. 执行工具，把结果拼回上下文
5. 让模型基于工具结果继续回答

关键点：要做好错误处理——模型输出的JSON可能格式不对、参数缺失、工具不存在，都要兜底。

### 4. Agent内的上下文怎么管理？

**答案**：

三层策略：

**短期记忆（对话历史）**：
- 保留最近N轮对话（通常5-10轮）
- 按token数截断（比如保留最近4000 token）
- 工具调用的中间结果也要保留

**工作记忆（当前任务）**：
- 当前正在处理的任务描述
- 已经收集到的信息
- 还需要获取的信息

**长期记忆（可选）**：
- 用向量数据库存储历史对话摘要
- 需要时检索相关历史

关键问题：上下文太长会稀释重点，太短会丢失信息。需要根据任务类型调整。

## 二、RAG（高频）

### 5. 检索的召回率和准确率如何？

**答案**：

**召回率（Recall）**：相关文档被检索到的比例
- 公式：检索到的相关文档数 / 所有相关文档数
- 召回率高 = 不漏掉相关文档

**准确率（Precision）**：检索到的文档中相关的比例
- 公式：检索到的相关文档数 / 检索到的所有文档数
- 准确率高 = 不混入无关文档

**Trade-off**：
- 放宽检索条件（比如降低相似度阈值）→ 召回率↑，准确率↓
- 收紧检索条件 → 准确率↑，召回率↓

**实际做法**：
- 先保证召回率（宁可多召回，不能漏）
- 再用Reranker重排序，提高准确率
- 最后用Top-K截断，只把最相关的喂给模型

### 6. 数据清洗和去重怎么做？

**答案**：

**文本清洗**：
- 去除HTML标签、特殊字符、多余空白
- 统一编码（UTF-8）
- 处理乱码、截断内容

**去重**：
- 精确去重：内容hash（MD5/SHA256）
- 模糊去重：语义相似度（embedding距离 < 阈值）
- 标题+首段去重：很多文章换个标题内容一样

**分块策略**：
- 按句子切分：语义完整，但块太小
- 按段落切分：块大小适中，但段落长短不一
- 固定长度切分：块大小一致，但可能切断语义
- **推荐**：固定长度 + overlap（比如512 token，overlap 50 token）

### 7. 有没有做过检索效果评估？

**答案**：

**评估方法**：

1. **人工标注**：准备一批问题和标准答案，标注哪些文档是相关的
2. **自动指标**：
   - MRR（Mean Reciprocal Rank）：第一个相关文档排在第几位
   - NDCG（Normalized Discounted Cumulative Gain）：排序质量
   - Hit Rate：Top-K里有没有相关文档
3. **端到端评估**：最终回答是否正确（不只看检索，还看生成）

**实操**：
- 先跑一批测试集，看指标
- 分析bad case：是检索没召回，还是召回了但排序靠后
- 针对性优化：调embedding模型、调分块策略、加Reranker

## 三、评测（中频）

### 8. 如何评估Agent的回答质量？

**答案**：

**评估维度**：

| 维度 | 说明 | 方法 |
|------|------|------|
| 正确性 | 答案是否正确 | 和标准答案对比 |
| 完整性 | 是否覆盖所有要点 | 检查要点覆盖率 |
| 相关性 | 是否回答了问题 | 人工判断 |
| 格式 | 是否符合要求 | 规则检查 |

**评估方法**：

1. **人工评测**：最准但慢、贵
2. **LLM-as-Judge**：用GPT-4打分，成本中等
3. **自动化指标**：
   - BLEU、ROUGE：和标准答案的n-gram重叠
   - 精确匹配：答案完全一致
   - F1：token级别的精确率和召回率

**关键**：没有单一指标能衡量所有维度，要组合使用。

### 9. 有没有用过自动化评测？

**答案**：

**工具**：

1. **RAGAS**：RAG专用评测框架
   - Faithfulness（忠实度）：回答是否基于检索到的内容
   - Answer Relevancy（答案相关性）：回答是否和问题相关
   - Context Precision（上下文准确率）：检索内容是否相关

2. **DeepEval**：通用LLM评测
   - 支持多种指标：G-Eval、Bias、Toxicity
   - 可以自定义评测维度

3. **自建pipeline**：
   - 批量跑测试集
   - 调用LLM打分
   - 统计指标、生成报告

**实操建议**：先用RAGAS快速验证，再根据业务需求定制评测维度。

## 四、模型微调（低频但加分）

### 10. 有没有微调经验？

**答案**：

**常见方法**：

| 方法 | 资源消耗 | 效果 | 适用场景 |
|------|----------|------|----------|
| 全量微调 | 高 | 最好 | 数据充足、资源充足 |
| LoRA | 低 | 接近全量 | 资源有限、快速迭代 |
| QLoRA | 更低 | 略低于LoRA | 极度资源受限 |

**关键点**：

1. **数据质量 > 数据数量**：1000条高质量数据 > 10000条低质量数据
2. **学习率**：通常1e-5到5e-5，太大会震荡，太小会欠拟合
3. **Epoch数**：通常2-5轮，太多会过拟合
4. **过拟合判断**：训练loss下降但验证loss上升

### 11. SFT和RLHF的区别？

**答案**：

**SFT（监督微调）**：
- 用标注数据直接训练
- 数据格式：(输入, 输出) 对
- 模型学习"模仿"标注数据

**RLHF（人类反馈强化学习）**：
- 用人类偏好训练奖励模型
- 流程：收集偏好数据 → 训练奖励模型 → 用PPO优化策略模型
- 模型学习"最大化奖励"

**关系**：SFT是RLHF的前置步骤。完整流程：SFT → Reward Model → PPO

**为什么需要RLHF**：
- SFT只能学习"模仿"，RLHF能学习"偏好"
- 有些任务没有标准答案，但人类能判断好坏
- RLHF能让模型输出更符合人类期望

## 五、Prompt工程（中频）

### 12. Prompt怎么设计？

**答案**：

**五要素**：

1. **角色设定**：你是一个专业的数据分析师...
2. **任务描述**：请帮我分析以下数据...
3. **输出格式**：以JSON格式输出，包含以下字段...
4. **示例**：输入xxx，输出xxx（Few-shot）
5. **约束**：不要编造数据，只基于提供的信息回答

**设计原则**：
- 清晰：不要有歧义
- 具体：不要模糊描述
- 结构化：用分隔符区分不同部分
- 示例：复杂任务一定要给示例

### 13. 有没有Prompt优化经验？

**答案**：

**方法**：

1. **A/B测试**：准备多个Prompt，跑同一批数据，对比效果
2. **错误分析**：看bad case，分析是Prompt哪里没说清楚
3. **迭代优化**：改一点 → 测试 → 看效果 → 再改

**自动优化工具**：
- **DSPy**：用代码定义Prompt，自动优化
- **OPRO**：用LLM自己优化Prompt
- **PromptFlow**：微软的Prompt工程工具

**关键**：不要追求"完美Prompt"，够用就行。Prompt优化是锦上添花，数据和模型才是根本。

## 六、工程基础（穿插考察）

### 14. Python基础

**高频考点**：

**装饰器**：
```python
def timer(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"耗时: {time.time() - start}s")
        return result
    return wrapper
```

**生成器**：
```python
def fibonacci():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b
```

**GIL**：
- Python的全局解释器锁，同一时刻只能执行一个线程
- CPU密集型任务用多进程，IO密集型任务用多线程/协程

### 15. 数据结构与算法

**高频题型**：
- 链表：反转、合并、环检测
- 树：遍历、深度、路径和
- 图：BFS、DFS、最短路径
- 动态规划：背包、最长子序列
- 排序：快排、归并、堆排

**LeetCode**：中等难度为主，偶尔困难

### 16. 系统设计

**问答系统设计**：

```
用户输入 → 意图识别 → 检索 → 生成 → 输出
              ↓
         工具调用（可选）
```

**关键组件**：
- 负载均衡：Nginx/ALB
- 缓存：Redis（热点问题缓存）
- 消息队列：Kafka/RabbitMQ（异步处理）
- 数据库：向量数据库（Milvus/Pinecone）+ 关系型数据库

## 七、面试心态与策略

**核心教训**：

1. **面试官要的是"你做过"，不是"你知道"**
   - 不要背概念，要讲实战
   - 讲清楚Why、What、How、Result

2. **不懂的不要硬撑**
   - 诚实说不了解比胡扯强
   - 可以说"这个我没做过，但我的思路是..."

3. **被追问细节是好事**
   - 说明面试官对你的项目感兴趣
   - 准备好：数据量、指标、踩坑、解决方案

**准备建议**：

- 把项目拆成模块，每个模块能讲10分钟
- 准备好被追问：为什么选这个方案？有没有更好的？
- 复习基础：算法+操作系统+网络
