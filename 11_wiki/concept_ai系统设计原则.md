# AI系统设计原则

---

## 一、何时真正需要多 Agent

> **2026-03-30 · Engineering Notes · Agent Systems**

### 核心命题

触发多 Agent 的条件只有一个：**状态空间需要隔离**。

其余所有判断标准都是这个命题的具体投影。其他情况下，**Single Agent + Memory System 更优**。

---

### 三条判断标准

|场景|判断标准|技术实现|本质|
|---|---|---|---|
|Context Protection|子任务产生 >1000 tokens 且大部分无关主任务|Context Isolation|隔离**读状态**|
|Parallelization|任务可独立分解，子任务间无依赖|Async Concurrency|隔离**写状态**|
|Specialization|工具集 >20 或跨领域，工具间互相干扰推理|Focused Toolsets|隔离**工具状态**|

## 二、Agent系统的四层机制：提示词、上下文、外部记忆与微调

### 核心观点

一个成熟的 Agent 系统通常同时依赖四层机制：

- **提示词 / 系统指令**：约束当前回合的角色、格式、安全规则和任务目标；
- **上下文 / 会话状态**：承载短期工作记忆，如最近几轮对话和当前中间结果；
- **外部记忆 / RAG / 工具**：提供可更新、可追溯、可按权限读取的信息；
- **微调 / 参数化能力**：固化长期稳定的行为模式、风格、格式和窄任务技能。

它们不是相互替代的关系，而是分工协作。

### 四层机制的分工

| 层次 | 主要作用 | 最适合存放的内容 |
|---|---|---|
| 提示词 / 系统指令 | 约束当前行为 | 角色、规则、输出格式、当前目标 |
| 上下文 / 会话状态 | 提供短期工作记忆 | 最近对话、任务进度、临时推理结果 |
| 外部记忆 / RAG / 工具 | 提供可更新的信息 | 文档、用户画像、业务数据、历史事件 |
| 微调 / 参数化能力 | 固化长期稳定模式 | 风格、偏好、格式、特定任务能力 |

### 微调的真实作用

微调最适合做的事是把 **稳定、重复、长期有效** 的模式写进模型行为里，例如：

- 固定输出格式；
- 特定语气和风格；
- 某类任务的稳定响应模式；
- 某些 instruction-following 问题的纠正。

**微调的价值主要在于**：

- 减少每次请求里重复携带的大量示例；
- 缩短提示词；
- 在固定任务上提升一致性；
- 让更小、更便宜的模型适配特定任务。

但微调通常 **不适合** 存放频繁变化的信息，例如：

- 某个用户昨天刚更新的偏好；
- 某次会话里的临时状态；
- 会不断变化的业务知识库；
- 需要可删除、可追溯、可审计的信息。

这些信息更适合放在外部记忆系统里。

### 什么是"模型记忆"

从工程角度看，模型记忆至少有两类：

- **参数记忆（parametric memory）**：模型权重中压缩的知识、模式和偏好；
- **非参数记忆（non-parametric memory）**：通过检索库、会话历史、工具系统、用户记忆存储提供的外部状态。

两者都可以被视为"记忆"，但适用场景不同：

- 参数记忆更适合长期稳定、不常变化的模式；
- 非参数记忆更适合可更新、可控制、可追溯的信息。

### 为什么今天的 Agent 仍然显得冗余

很多 Agent 系统之所以冗余，不是因为没有"真正记忆"，而是因为系统需要同时完成这些工程任务：

- 管理会话状态；
- 压缩和摘要长上下文；
- 检索外部知识；
- 维护工具调用链；
- 保证安全和输出约束。

因此，现实里的优化方式通常不是单靠微调，而是多种方法配合：

1. 用更短、更清晰的系统提示词约束行为。
2. 对长对话做摘要和状态压缩。
3. 把经常变化的知识放进外部检索库。
4. 把强个性化信息放进用户级记忆存储。
5. 只把高频、稳定、可泛化的模式交给微调。
6. 在高重复输入场景下使用提示词缓存（prompt caching）。

### 关于会话 ID 和长期记忆

从用户体验上看，一个系统完全可以只暴露 `session ID`，然后自动恢复上下文。

但在系统内部，这通常仍然意味着：

- 读取历史消息或摘要；
- 从记忆库检索相关信息；
- 把必要状态重新组织进当前请求；
- 或通过工具把外部信息提供给模型。

所以，`session ID` 更接近 **状态入口**，而不是模型天然拥有长期、可写记忆的证明。

### 更稳妥的一句话总结

> **权重负责长期稳定的能力倾向，外部记忆负责可更新的事实与经历，上下文负责当前任务的工作记忆，提示词负责当下约束。**

## 三、何时不需要 AI：决策层级

> 来源：十年老技术开发的 AI Agent 探索之路（2026-05-07）

**原则：代码优先于 Prompt，能用 10 行 Bash 解决的，别折腾 AI。**

拿到新需求时，从底行往上看——先问"10 行 Bash 能搞定吗？"，再问"一次 LLM 调用够吗？"，最后才考虑 Agent。

| 层级 | 适用场景 | 示例 | 不确定性 |
|------|---------|------|---------|
| 目标层 | 想清楚到底要解决什么 | 想清楚后发现不需要写代码 | 最低 |
| 代码层 | 确定性逻辑 | if/else、正则、模板引擎 | 低 |
| CLI 层 | 组合现有工具 | grep + jq + curl | 中低 |
| Prompt 层 | 需要语义理解和判断 | 需求翻译、文案生成 | 中高 |
| Agent 层 | 多步推理、动态决策 | 自动修 bug、端到端流程 | 最高 |

每往上一层，不确定性增加一个量级，成本也增加一个量级。

> 这不是反 AI，是尊重工程。很多项目明明 cron + curl 就能搞定的定时数据采集，非要套 LangChain + Agent 循环 + tool calling，最后效果还不如写死的脚本稳定。
