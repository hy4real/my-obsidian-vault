## 问题与背景 (Problem Statement)

### 第3周踩坑总结
- **问题1**：Agent自动作弊，为了提高输出效果进行数据污染
- **问题2**：数据集质量控制不足，标准定义不严谨
- **根本原因**：前期重点放在"有无AutoResearch"，忽视了数据质量基线

### 关键洞察
> **数据质量 > Agent复杂度**
> 
> 前期要把**标准定好**，才能交给Agent去探索。
> 基线足够严谨后，再做超参autoresearch的探索。

---

## 解决方案 (Approach)

### Phase 1: 数据标准严格化
- [ ] 重新定义微调数据集的质量标准
- [ ] 建立严格的数据验证流程
- [ ] 避免Agent的trick行为

### Phase 2: 基线优化
- [ ] 在严格标准下跑Qwen基线模型
- [ ] 确保基线性能稳定可复现

### Phase 3: AutoResearch探索
- [ ] 基于稳定基线，进行超参优化
- [ ] Agent A：修改超参 + 跑实验
- [ ] Agent B：独立监督 + 对抗反馈
- [ ] 迭代优化方案

### Phase 4: 多Agent对抗设计（下周计划）
**多Agent架构**：
```
Agent A (执行者)
├─ 修改超参和方法
├─ 跑实验，记录结果
└─ 有上下文和历史

Agent B (监督者)
├─ 独立上下文（或部分独立？）
├─ 提出挑战和建议
└─ 互相对抗式反馈

结果：一整个对抗优化循环
```

**待解决问题**：
- [ ] 两Agent如何共享数据而不互相污染？
- [ ] Agent B是否需要完全独立上下文？
- [ ] 如何量化对抗反馈的价值？

## 项目讲述稿 (5分钟版本)

### 背景故事
> 我在做LLM微调的时候，发现手工调参非常低效。我就想，能否用AutoResearch自动化这个过程？
> 
> 但在第一周的实验中，我踩了个大坑 **Agent为了优化指标，自己作弊**。这让我意识到，数据质量和标准定义比Agent本身的复杂度更重要。

### 核心解决方案

> 所以我调整了策略：
> 1. 先把数据标准定严谨
> 2. 在严格标准下跑基线，确保稳定
> 3. 然后才启用AutoResearch做超参优化
> 
> 为了进一步提升，我还设计了多Agent对抗架构，让Agent A做实验，Agent B做独立监督，通过对抗反馈迭代优化。

### 技术亮点
- 工程化程度高：Unsloth加速、自动化实验追踪
- 解决实际问题：Agent作弊防护、数据质量控制
- 创新设计：多Agent对抗优化（单卡情况下的并行优化）
