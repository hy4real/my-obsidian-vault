# Karpathy 知识库管理新玩法 - LLM + Obsidian + Wiki

> [!info] 文章信息
> **来源**: 微信公众号 - 逛逛 GitHub  
> **阅读时间**: 15 分钟  
> **发布时间**: 未知（推荐查看原文）  
> **链接**: [原文](https://mp.weixin.qq.com/s/FG2gEdgZwpMkA3ApQMksnw)

## 核心观点

**核心理念**: 不要把 LLM 当搜索引擎用，而是让它像程序员写代码一样，帮你**持续维护一个 Markdown 知识库**。用户负责找资料和提问，LLM 负责总结、交叉引用、分类整理、保持一致性。知识库如同滚雪球般越滚越大，形成**可复利增长的资产**。

---

## 传统方式的问题

传统 RAG（检索增强生成）方案存在致命缺陷：
- **没有积累**: 每次提问时 LLM 都从原始文档重新检索和拼接
- **重复计算**: 复杂问题需要综合多篇文档，但每次提问都要重新推导
- **知识流失**: 问完即弃，答案不会被保存回库中
- **缺乏沉淀**: 信息从未被真正结构化和优化

**例**: NotebookLM、ChatGPT 文件上传、大多数 RAG 系统都属于这类模式。

### Karpathy 的新方案

关键改变：让 LLM **持续地、增量式地构建和维护一个 Wiki**（结构化的、互相链接的 Markdown 文件集合）。
- 新资料添加时，LLM 不仅索引它，还会**整合进现有 Wiki**
- **更新相关实体页面、修正主题摘要、标注新旧矛盾**
- 知识编译一次，持续保持最新
- 交叉引用已建好，矛盾已标注，综合分析已反映全部内容

---

## 三层架构

### 第一层：Raw Sources（原始资料层）
- 收集的论文、文章、图片、数据文件
- 不可变，LLM 只读不写
- 是原始数据来源的保证

### 第二层：The Wiki（知识库层）
- LLM 生成的 Markdown 文件目录
- 包含：摘要、实体页面、概念页面、对比分析、综述等
- 完全由 LLM 拥有和维护，用户读它，LLM 写它

### 第三层：The Schema（规则文件）
- 告诉 LLM Wiki 的组织方式、约定、流程
- 对 Claude Code 就是 `CLAUDE.md`，对 Codex 就是 `AGENTS.md`
- 关键的配置文件，可持续迭代优化

---

## 三个核心操作

### ① Ingest（录入）
**流程**:
1. 往原始资料目录丢一个新文件
2. LLM 读资料、跟你讨论要点
3. 在 Wiki 中写摘要页、更新索引
4. 更新相关的实体和概念页面

**规模**: 一个来源可能牵动 10-15 个 Wiki 页面的更新

Karpathy 的做法：**一个一个录入，边录边看**，引导 LLM 重点关注什么。

### ② Query（提问）
**特点**:
- 对 Wiki 提问，LLM 搜索相关页面后综合回答
- 回答形式多样：Markdown、对比表格、Marp 幻灯片、matplotlib 图表等
- **关键洞察**: 好的回答可以回存到 Wiki，变成新页面
- 每次探索和提问都在持续丰富知识库——**知识复利增长**

### ③ Lint（体检）
**定期检查**:
- 页面之间的矛盾
- 被新资料取代的过时信息
- 没有入链的孤儿页面
- 提到但没有独立页面的重要概念
- 缺失的交叉引用

**LLM 还会**建议你应该研究什么新问题、找什么新资料，保证 Wiki **健康增长**。

## 实际工作流

### 工作环境设置
- 一边开 Agent（Claude Code、Codex 等）
- 一边开 Obsidian（实时浏览结果）
- LLM 根据对话编辑 Wiki，用户在 Obsidian 浏览

### 角色隐喻
> Obsidian 是 IDE，LLM 是程序员，Wiki 是代码库。

### 实用工具
- **Obsidian Web Clipper**: 快速把网页文章转成 Markdown
- **本地图片下载**: 让 LLM 直接查看和引用图片
- **index.md**: 内容目录
- **log.md**: 时间线日志

### 规模与工具选择
在中等规模下（约 100 个来源、几百个页面）：
- **直接靠索引文件就够用**，无需向量数据库或复杂 RAG 基础设施
- LLM 先读索引定位相关页面，再深入阅读具体内容，效果很好

## 为什么这个方案管用

### 人类的痛点
维护知识库最烦人的不是阅读和思考，而是**琐碎的簿记工作**：
- 更新交叉引用
- 保持摘要最新
- 标注新旧数据矛盾
- 维护页面间的一致性

**现象**: 人类维护负担增长速度 > 价值增长速度 → Wiki 荒废

### LLM 的优势
- **不知疲倦**: 不会厌倦，不会忘记更新交叉引用
- **批量修改**: 可一次修改 15 个文件
- **维护成本接近零**: Wiki 能一直保持健康

### 历史渊源
这个思路与 **1945 年 Vannevar Bush 的 Memex 构想**精神一脉相承：
- 个人的、精心策划的知识存储
- 文档间的关联和文档本身一样有价值
- **Bush 没解决**: 谁来干维护工作？
- **LLM 搞定了**: 自动维护

2. **Claude Code Skill**
   - 一行命令装上
   - 直接在 Claude Code 里 ingest 来源

   - 专为这种思维工作流设计的 IDE

## 如何上手

### 最简单的方式
将 Gist 内容复制给你的 Agent，让它帮你搭建。

**Gist 地址**: https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f

### 所需工具
**必需**:
- 一个 Agent（Claude Code、Codex、OpenCode 都行）
- Obsidian

**可选增强**:
- **qmd** 搜索引擎：Wiki 变大后提供更好的搜索能力
- **Marp** 格式：从 Wiki 内容直接生成幻灯片
- **Dataview 插件**：基于页面元数据做动态查询
- **图谱视图**：看 Wiki 整体形态（连接关系、枢纽、孤岛）

### 版本控制
整个 Wiki 本质上是 Git 仓库的 Markdown 文件：
- 版本历史、分支、协作都是现成的

## 最大的启发

### 思维方式转变
大多数人目前用 LLM 还是把它当搜索引擎或聊天机器人：
- 问一次 → 答一次 → 完了

**Karpathy 的思路**：把 LLM 当成**不知疲倦的知识工程师**
- 帮你把散乱信息编译成结构化知识网络
- 网络随使用不断增长
- 每次探索、每次提问都让知识库变得更好
- 这就是**复利的力量**

> 
> **核心思路**
> 
> Karpathy 提出的核心思路是：不要把 LLM 当搜索引擎用，而是让它像程序员写代码一样，帮你持续维护一个 Markdown 知识库。你负责找资料、提好问题，LLM 负责所有的脏活累活：总结、交叉引用、分类整理、保持一致性。你在 Obsidian 里浏览，LLM 在后台编辑，知识库就像滚雪球一样越滚越大。
> 
> **传统方式的问题**
> 
> 大多数人用 LLM 管文档的方式其实都差不多：把文件上传上去，每次提问的时候 LLM 检索相关片段，然后生成回答。这就是大家常说的 RAG 方案。NotebookLM、ChatGPT 的文件上传、大部分 RAG 系统都是这个套路。Karpathy 觉得这个方式有个致命问题：没有积累。每次提问，LLM 都得从原始文档里重新找、重新拼。你问一个需要综合五篇文档的复杂问题，它每次都得从头来。问完了，答案就没了，下次还得重新推导一遍。知识从来没有被真正沉淀下来。
> 
> **应该让知识复利增长**
> 
> Karpathy 的方案完全不一样。他让 LLM 不是每次从原始文档里检索，而是持续地、增量式地构建和维护一个 Wiki：一个结构化的、互相链接的 Markdown 文件集合。当你往里面添加一个新的资料来源，LLM 不是简单地把它索引进去等着以后被检索。它会读这个资料，提取关键信息，然后把信息整合进已有的 Wiki 里：更新相关实体的页面、修正主题摘要、标注新数据和旧结论的矛盾之处。知识编译一次，然后持续保持最新。这才是关键区别：Wiki 是一个持久的、可复利增长的资产。
> 
> **三个核心操作**
> 
> **Ingest（录入）**: 你往原始资料目录里丢一个新文件，让 LLM 处理。LLM 会读这个资料、跟你讨论要点、在 Wiki 里写一个摘要页、更新索引、更新相关的实体和概念页面。一个来源可能牵动 10 到 15 个 Wiki 页面的更新。Karpathy 说他喜欢一个一个录入，边录边看，引导 LLM 重点关注意什么。
> 
> **Query（提问）**: 你对着 Wiki 提问，LLM 搜索相关页面后综合回答。回答的形式可以很多样：Markdown 页面、对比表格、Marp 幻灯片、matplotlib 图表都行。这里面有一个重要洞察：好的回答可以回存到 Wiki 里，变成新的页面。这样你每次的探索和提问都在持续丰富知识库，知识在复利增长。
> 
> **Lint（体检）**: 定期让 LLM 对 Wiki 做健康检查。找页面之间的矛盾、被新资料取代的过时信息、没有入链的孤儿页面、提到但没有独立页面的重要概念、缺失的交叉引用。LLM 还很擅长建议你应该去研究什么新问题、找什么新资料。这一步保证 Wiki 在增长过程中保持健康。
> 
> **为什么这玩意儿管用**
> 
> 说实话维护知识库最烦人的事从来不是阅读和思考，而是那些琐碎的簿记工作。更新交叉引用、保持摘要最新、标注新旧数据矛盾、维护几十个页面之间的一致性。这些活儿人类做着做着就烦了，然后 Wiki 就慢慢荒废了。人类放弃 Wiki 的原因就是维护负担增长得比价值快。但 LLM 不会厌倦，不会忘记更新一个交叉引用，可以一次性修改 15 个文件。维护成本接近零，Wiki 就能一直保持健康。
> 
> **Agent 时代的实现**
> 
> 这个 Gist 发出去之后，各路开发者反应非常快，已经有一堆人在做具体实现了。有人用 Go 写了个叫 sage-wiki 的工具，一个二进制文件就能跑，支持增量编译、搜索、问答。还能当 MCP Server 暴露给任何 LLM Agent 用。有人做了个 Claude Code 的 Skill，一行命令就能装上，直接在 Claude Code 里 ingest 来源。还有人做了 Thinking-Space，一个专门为这种思维工作流设计的 IDE。
